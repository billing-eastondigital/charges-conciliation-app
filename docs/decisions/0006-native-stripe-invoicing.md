# ADR 0006 — Native Stripe Invoice Creation (replacing Make.com)

**Date**: 2026-07-02  
**Status**: Accepted  
**Author**: Marco / Claude

---

## Context

Until now, Stripe invoices were created via a Make.com scenario ("Integration Stripe"). That scenario:

1. **Triggered** on Airtable records where `Ready for Billing = TRUE` AND `Invoice Created? = FALSE`
2. **Validated** that at least one line item amount was > 0
3. **Created a Stripe invoice** (`collection_method: send_invoice`, `days_until_due: 1`, `auto_advance: false`) with the Memo field as description and a Footer field
4. **Added up to 5 line items** via parallel router branches (each gated: skip if amount = 0); amounts passed as cents (dollars × 100)
5. **Wrote back** to Airtable: `Invoice Created? = true` and `Stripe Invoice ID`
6. **Retrieved** the invoice for audit

Airtable was the source of truth for billing data. The recon app now owns that data in Supabase (`expected_charges`), making the Airtable dependency redundant and the Make.com scenario obsolete.

---

## Decision

Replace the Make.com scenario with a Supabase Edge Function `create-stripe-invoices`.

---

## Data model mapping

| Make.com / Airtable field | Supabase equivalent |
|---|---|
| `Stripe Customer ID` | `expected_charges.stripe_id` |
| `Ready for Billing = TRUE` | `expected_charges.ready_for_billing = true` |
| `Invoice Created? = FALSE` | `expected_charges.invoice_url IS NULL` |
| `Line Item N` / `Line Item N amount` | `expected_charges.billing_detail->'line_items'` (jsonb array `[{text, amount}]`) |
| `Memo` | `expected_charges.billing_detail->>'memo'` (ADS clients) or `account_name \|\| ' ' \|\| period_label \|\| ' Invoice'` (SUBSCRIPTION) |
| `Footer` | Hardcoded default (not stored yet — can be added later) |
| `Stripe Invoice ID` → Airtable | `expected_charges.stripe_invoice_id` (new column, see migration `20260702000001`) |
| `Invoice Created? = true` | `expected_charges.invoice_status = 'open'` |
| Invoice hosted URL | `expected_charges.invoice_url` |

---

## Key behavioral differences from Make.com

### 1. Grouping: one invoice per `stripe_id`, not per row

The Make scenario created one invoice per Airtable record. In Supabase, a single `stripe_id` may have **multiple `expected_charges` rows** in the same period (e.g., an ADS row + a SUBSCRIPTION add-on row for Cheap Dealer Supplies). The Edge Function groups by `stripe_id` and creates one Stripe invoice with all line items from all matching rows.

### 2. Line item source

ADS clients (`ADS_REVENUE`, `ADS_COST`): line items come from `billing_detail->'line_items'` — a jsonb array of `{text, amount}` objects generated by `generate_ads_billing()`.

SUBSCRIPTION clients: no structured line items in billing_detail. A single invoice item is created: description = `account_name`, amount = `expected_amount`.

### 3. Finalization

Like Make.com, invoices are created with `auto_advance: false` (draft). The owner reviews and manually finalizes/sends from the Stripe Dashboard, or a future "Send Invoice" action in the app can call `stripe.invoices.finalizeInvoice()`.

### 4. Write-back

After creation, the function updates all `expected_charges` rows for that `stripe_id` + `period_label` with:
- `stripe_invoice_id` — Stripe invoice ID (`in_…`)
- `invoice_url` — Stripe hosted invoice URL
- `invoice_status = 'draft'` (draft until finalized)

This mirrors the Make write-back to Airtable but uses the existing `invoice_url` / `invoice_status` columns plus the new `stripe_invoice_id` column.

---

## Trigger

The Edge Function is invoked:
- **Manually** via the billing page UI — "Create Invoices" button for a period
- **Optionally** per-client via a row action
- (Future) automatically after `reconcile-period` for rows already marked `ready_for_billing`

It is NOT triggered by a scheduled cron — invoice creation is always an intentional owner action.

---

## Consequences

- Make.com scenario can be deactivated once the Edge Function is deployed and tested.
- Airtable is no longer needed for the invoicing workflow.
- `stripe_invoice_id` column must be added to `expected_charges` (migration `20260702000001`).
- The billing page UI needs a "Create Invoice" action and invoice status display (separate task).
